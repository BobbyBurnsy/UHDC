IT Pro Summary

This is a targeted intelligence engine designed to locate a specific user on the network when their historical data is stale. It utilizes a cascading fallback logic: It first resolves the exact AD identity, checks the central UserHistory.json for a quick WMI match, and if the user has moved desks, it pivots using their Active Directory "Office" or "OU" attribute to perform a highly restricted, context-aware sweep of their department's computers.

When to Use This Tool

The Scenario

You are on a live call with a user who needs immediate remote assistance. You run a Smart Search, but their last known laptop is turned off. The user explains they are hot-desking today or using a shared kiosk in their department, but they don't know how to find the computer name. You cannot remote in without a hostname.

The Fix

Instead of walking the user through opening the command prompt to type hostname, the technician clicks "Net Scan." The script realizes the user's old laptop is offline. It looks at their AD profile, sees they work in "Accounting," and instantly pulls a list of the 40 computers located in the Accounting OU. It rapidly pings and queries those 40 machines in the background. Within 10 seconds, it finds the user logged into DESKTOP-HQ-9912, updates the central database, and auto-fills the console so the technician can immediately send a Remote Assistance invite.

[TRAINING] STEP 1: RESOLVE AD IDENTITY

WHEN TO USE THIS:
This happens automatically when you click 'Net Scan'. It ensures we are searching for the exact, correct user account before we start sweeping the network.

WHAT IT DOES:
We query Active Directory for the provided username to retrieve their exact 'SamAccountName', 'Office' code, and 'DistinguishedName' (OU path). We will use this data to narrow down our search radius later.

IN-PERSON EQUIVALENT:
Opening ADUC, searching for the user, and checking the 'Organization' tab to see what building or department they work in so you know which floor to walk to.

$exact = Get-ADUser -Identity $TargetUser -Properties Office, DistinguishedName
$ResolvedUser = $exact.SamAccountName
[TRAINING] STEP 2: CHECK HISTORICAL DATABASE

WHEN TO USE THIS:
Before we waste time scanning hundreds of computers, we check if we already know where this user usually sits.

WHAT IT DOES:
We read the central 'UserHistory.json' file to find the last known PC this user logged into. We then send a quick WMI query to that specific PC to see if they are still there. If they are, the scan finishes instantly.

IN-PERSON EQUIVALENT:
Checking your personal notes or asking a coworker, 'Hey, doesn't John usually sit at desk 4B?' and walking straight there first.

$match = $raw | Where-Object { $_.User -eq $ResolvedUser } | Sort-Object LastSeen -Descending | Select-Object -First 1
$check = Get-CimInstance -ClassName Win32_ComputerSystem -ComputerName $match.Computer
[TRAINING] STEP 3: CONTEXT-AWARE TARGETING

WHEN TO USE THIS:
If the user isn't at their usual desk (or they are a new hire with no history), we need to scan the network. But scanning 5,000 PCs takes too long.

WHAT IT DOES:
We use the 'Office' attribute or the Organizational Unit (OU) we gathered in Step 1 to pivot our search. If the user is in the 'Accounting' OU, we only pull a list of computers that are also in the 'Accounting' OU. This reduces our scan target from thousands of PCs down to just a few dozen.

IN-PERSON EQUIVALENT:
Knowing the user works in Accounting, so you only walk the floor of the Accounting department looking for them, rather than checking every desk in the entire building.

if ($dn -match "OU=Users,(.+)$") {
    $searchBase = "OU=Computers,$($matches[1])"
    $computers = Get-ADComputer -Filter * -SearchBase $searchBase
}
[TRAINING] STEP 4: EXECUTE WMI SWEEP

WHAT IT DOES:
We iterate through our narrowed list of computers. We send a 500ms ping to see if the PC is turned on. If it is, we use WMI to ask the PC 'Who is logged in right now?'. As soon as we find a match for our target user, we break the loop, update the history database, and send the PC name back to the GUI.

IN-PERSON EQUIVALENT:
Walking down the row of desks in the Accounting department, wiggling the mouse on every active computer, and reading the lock screen until you find the user you are looking for.

foreach ($pc in $computers) {
    if (Test-Connection $pc) {
        $compInfo = Get-CimInstance Win32_ComputerSystem -ComputerName $pc
        if ($compInfo.UserName -match $ResolvedUser) { break }
    }
}

Technical Execution Pipeline

  1. Identity Resolution (LDAP): Queries AD over Port 389 to resolve the exact SamAccountName and extract the user's DistinguishedName and Office attributes.
  2. History Verification (SMB/WMI): Reads UserHistory.json over Port 445. If a previous PC is found, it sends a targeted WMI query over Port 5985 to verify if the user is still logged in there.
  3. Subnet Pivoting (LDAP): If the history check fails, it uses RegEx to parse the user's OU (e.g., OU=Users,OU=Accounting,DC=Domain) and dynamically rewrites it to target the corresponding computer OU (OU=Computers,OU=Accounting,DC=Domain). It queries AD for all computer objects within that specific scope.
  4. Rapid Polling (ICMP/WMI): Iterates through the narrowed list. It uses a highly aggressive 500ms ICMP ping timeout to quickly skip offline machines, followed by a Win32_ComputerSystem WMI query to check the active console user.
  5. Database Injection & UI Handoff: Upon a successful match, it breaks the loop, silently triggers Helper_UpdateHistory.ps1 to log the new location, and outputs the [GUI:UPDATE_TARGET:...] tag to auto-fill the technician's dashboard.

Q&A

Q: Why doesn't the script just scan the entire Active Directory environment?
Performance and safety. Scanning 5,000 computers sequentially via WMI could take 15 to 30 minutes, which is useless when a user is waiting on the phone. Furthermore, aggressive global WMI sweeps can trigger intrusion detection systems (IDS) or be flagged as lateral movement by security teams. By pivoting off the user's OU, we reduce the scan target to a highly localized, logical subnet, returning results in seconds.
Q: What happens if the user's AD profile doesn't have an Office attribute or a clean OU structure?
The script is designed to fail safely. If it cannot determine a logical, narrowed scope to scan, it will abort the operation and display: [X] ABORTING: User context (Office/OU) not found. Global Scanning is disabled for performance/safety. This prevents the script from accidentally initiating a domain-wide sweep.
Q: Does this tool find users who are connected via VPN or Remote Desktop?
Yes. The Win32_ComputerSystem WMI class registers the primary user logged into the Windows session, regardless of whether they are physically at the keyboard or connected via a corporate VPN tunnel, as long as the endpoint is reachable over the network.
