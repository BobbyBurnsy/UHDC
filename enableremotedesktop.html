IT Pro Summary

This is a multi-layered configuration tool designed to restore administrative access to a remote endpoint. It utilizes WinRM to systematically reconfigure the three core components required for Remote Desktop Protocol (RDP) connections: modifying the system registry to allow connections, punching a hole in the local Windows Defender Firewall for Port 3389, and ensuring the underlying Terminal Service is actively running.

When to Use This Tool

The Scenario

You are on a live call with a user who needs hands-on assistance. You attempt to connect to their machine using Microsoft Remote Assistance (MSRA) or standard RDP, but the connection is actively refused or times out. The user's machine is online, but the Remote Desktop feature was either disabled by a recent Windows Update, turned off manually, or the local firewall profile shifted and blocked the port. Walking a non-technical user through the Control Panel to re-enable these settings is time-consuming and frustrating.

The Fix

While still talking to the user, the technician clicks "Enable RDP" in the UHDC. In less than 3 seconds, the script forces the registry key open, applies the correct firewall rules, and starts the service. The technician immediately re-sends the MSRA invite, the connection succeeds, and the support session continues without the user ever having to open their system settings.

Interactive Training Engine

When a junior technician runs this tool with Training Mode enabled, the UHDC pauses execution at each critical phase. It displays a pop-up explaining why the action is necessary, what the in-person equivalent would be, and reveals the exact PowerShell code being executed under the hood. Here is what the trainee sees during an RDP Enablement:

[TRAINING] STEP 1: ENABLE RDP IN REGISTRY

WHEN TO USE THIS:
Use this when you need to establish a Remote Desktop connection to a PC, but the feature is currently disabled in the system settings.

WHAT IT DOES:
We are remotely modifying the 'fDenyTSConnections' registry key. Changing this value from 1 (Deny) to 0 (Allow) tells Windows to accept incoming Terminal Services (RDP) connections.

IN-PERSON EQUIVALENT:
Open System Properties (sysdm.cpl) > Remote tab > Select 'Allow remote connections to this computer'.

Invoke-Command -ComputerName $Target -ScriptBlock { Set-ItemProperty -Path 'HKLM:\System\CurrentControlSet\Control\Terminal Server' -Name 'fDenyTSConnections' -Value 0 }
[TRAINING] STEP 2: OPEN WINDOWS FIREWALL

WHEN TO USE THIS:
Use this when RDP is enabled in the registry, but connections are still timing out because the local Windows Defender Firewall is blocking port 3389.

WHAT IT DOES:
We are using the NetSecurity module to enable the predefined 'Remote Desktop' firewall rule group across the active network profiles.

IN-PERSON EQUIVALENT:
Open Windows Defender Firewall > 'Allow an app or feature through Windows Defender Firewall' > Check the box for 'Remote Desktop'.

Invoke-Command -ComputerName $Target -ScriptBlock { Enable-NetFirewallRule -DisplayGroup 'Remote Desktop' }
[TRAINING] STEP 3: START TERMINAL SERVICE

WHEN TO USE THIS:
Use this to ensure the underlying Remote Desktop Service is actually running and actively listening for connections.

WHAT IT DOES:
We are configuring the 'TermService' to start automatically on boot, and then forcefully starting it right now.

IN-PERSON EQUIVALENT:
Open Services (services.msc), locate 'Remote Desktop Services', right-click and select Properties, change Startup type to 'Automatic', and click 'Start'.

Invoke-Command -ComputerName $Target -ScriptBlock { Set-Service -Name 'TermService' -StartupType Automatic; Start-Service -Name 'TermService' }

Technical Execution Pipeline

  1. Pre-Execution Validation: Executes a fast ICMP Ping to verify endpoint connectivity before attempting WinRM connections.
  2. Registry Modification (WinRM): Establishes a remote PowerShell session over Port 5985 (HTTP). It uses Set-ItemProperty to target the HKLM:\System\CurrentControlSet\Control\Terminal Server path, flipping the fDenyTSConnections DWORD from 1 to 0.
  3. Firewall Configuration (WinRM): Re-uses the session to execute Enable-NetFirewallRule, targeting the native Remote Desktop display group to open TCP/UDP Port 3389.
  4. Service Initialization (WinRM): Executes Set-Service to ensure the TermService is set to Automatic startup, followed by Start-Service to immediately bind the listening port.
  5. Audit Logging: Logs the successful execution to the central UHDC CSV database via SMB (Port 445).

Q&A

Q: Why do we need to do all three steps? Isn't changing the registry enough?
No. Windows security is compartmentalized. Changing the registry tells the OS it is allowed to accept connections, but if the Windows Firewall is actively blocking Port 3389, the connection will still drop. Furthermore, if the underlying Terminal Service is stopped, there is no process listening on that port anyway. This script provides a holistic fix to ensure all three dependencies are met.
Q: Does this bypass Network Level Authentication (NLA)?
No. This script only enables the listening port and the service. It does not modify the UserAuthentication registry key. If your domain enforces Network Level Authentication for RDP connections via Group Policy, that security standard remains perfectly intact.
Q: Will clicking this button kick the user off their computer?
No. Enabling the feature happens silently in the background. However, if you subsequently use a standard RDP client (mstsc.exe) to connect to a Windows 10/11 workstation, it will prompt the user to allow you to take over their session, locking their screen. For live support where you want to share the screen with the user, you should use Microsoft Remote Assistance (MSRA) or SCCM Remote Control after enabling this feature.
