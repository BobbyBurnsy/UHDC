IT Pro Summary

This is a background telemetry and asset-mapping engine. It enumerates the Active Directory environment for supported workstations, polls them concurrently to identify active user sessions, and compiles a centralized, highly resilient JSON database. This creates an instant "User-to-Computer" lookup table for the help desk, utilizing an additive composite key structure to accurately track users across multiple devices without data overwrites.

When to Use This Tool

The Scenario

Active Directory natively lacks a fast, reliable way to show which specific workstation a user is currently logged into. When a user calls the help desk or submits a vague ticket like "My computer is acting up," technicians waste valuable minutes trying to walk the user through finding their hostname sticker or navigating system settings before they can even connect.

The Fix

GlobalNetworkMap.ps1 runs on a weekly schedule to silently sweep the network and update the master JSON tracking database. Now, when a technician pulls up the Active Directory intelligence search within the UHDC, the console cross-references this database to display the active computer name right alongside the user's AD profile. The technician gets the exact hostname instantly, completely bypassing the "How do I find my computer name?" conversation.

[TRAINING] STEP 1: LOAD EXISTING DATABASE

WHEN TO USE THIS:
This tool is restricted to Master Admins and is typically run twice a week (e.g., Mondays and Thursdays at 10 AM) to build and maintain the global asset map used by the Smart User Search.

WHAT IT DOES:
We are loading the central 'UserHistory.json' database into memory. We use a composite key ('User-Computer') to ensure that if a user logs into a second laptop, it adds a new record rather than overwriting their primary desktop.

IN-PERSON EQUIVALENT:
Opening a master Excel spreadsheet on a shared network drive that tracks which employee is assigned to which physical desk or computer.

$raw = Get-Content $HistoryFile -Raw | ConvertFrom-Json
foreach ($entry in $raw) { $masterDB["$($entry.User)-$($entry.Computer)"] = $entry }
[TRAINING] STEP 2: QUERY ACTIVE DIRECTORY FOR WORKSTATIONS

WHAT IT DOES:
We are querying Active Directory for all enabled computer objects running Windows 10 or Windows 11. This LDAP filter ensures we only target active client endpoints, filtering out servers, disabled PCs, and stale objects so we don't waste time scanning them.

IN-PERSON EQUIVALENT:
Opening Active Directory Users and Computers (ADUC), creating a custom saved query for 'Operating System starts with Windows 10', and exporting the list to a CSV file to know which desks to check.

$filter = "Enabled -eq 'true' -and (OperatingSystem -like '*Windows 10*' -or OperatingSystem -like '*Windows 11*')"
$computers = Get-ADComputer -Filter $filter | Select-Object -ExpandProperty Name
[TRAINING] STEP 3: PING SWEEP & WMI USER QUERY

WHAT IT DOES:
For every computer found in AD, we send a fast ping. If it responds, we establish a WMI connection to query the 'Win32_ComputerSystem' class and extract the 'UserName' property to see who is currently logged in. We then update their 'LastSeen' timestamp in our memory dictionary.

IN-PERSON EQUIVALENT:
Walking the floor, going desk to desk, wiggling the mouse on every active computer, and writing down the username displayed on the lock screen.

if (Test-Connection -ComputerName $pc -Count 1 -Quiet) {
    $compInfo = Get-CimInstance -ClassName Win32_ComputerSystem -ComputerName $pc
    $rawUser = $compInfo.UserName
}
[TRAINING] STEP 4: ATOMIC DATABASE SAVE

WHAT IT DOES:
We are converting our updated memory dictionary back into JSON format. To prevent database corruption if the script crashes or the network drops mid-save, we write the data to a '.tmp' file first, and then instantly swap it with the live 'UserHistory.json' file.

IN-PERSON EQUIVALENT:
Saving your updated Excel tracker as 'Tracker_New.xlsx', deleting the old 'Tracker.xlsx', and renaming the new file to replace it.

Set-Content -Path $TempFile -Value $jsonOutput -Force

Technical Execution Pipeline

  1. Database Initialization (SMB): Connects to the central network share via Port 445 (SMB). Evaluates the health of the existing UserHistory.json database. If it passes the byte-length health check, it safely duplicates it to a .bak file before making any changes.
  2. Directory Enumeration (LDAP): Uses the Get-ADComputer cmdlet to query Active Directory over Port 389 (LDAP) or Port 636 (LDAPS). It utilizes a strict -Filter to isolate only active, enabled endpoints running Windows 10 or Windows 11, deliberately excluding servers and deprecated operating systems from the scan loop.
  3. Endpoint Polling (ICMP & WSMan): Iterates through the AD array. First, it uses an ICMP ping to verify the endpoint is online. If responsive, it executes a Get-CimInstance query against Win32_ComputerSystem over Port 5985 (WSMan/WinRM) to extract the currently logged-on user.
  4. Data Transformation & Logic: Cleans the raw domain strings (DOMAIN\User becomes User) and generates a composite key ($cleanUser-$pc). This explicitly prevents a user who logs into a conference room PC from overwriting their primary laptop record in the database.
  5. Atomic Auto-Saving (SMB): To protect against network drops or script termination, it executes an atomic save operation every 50 loops over Port 445 (SMB). It serializes the array to a .tmp file first, confirms the JSON payload is valid, and only then overwrites the live production .json file.

Q&A

Q: What network ports are required for this engine to map the network?
This script is network-intensive. It requires Port 389/636 (LDAP) to query Active Directory for the computer list. It uses Port 5985 (WSMan/HTTP) to query WMI on the target endpoints for the logged-on user. Finally, it uses Port 445 (SMB) to read, backup, and save the central JSON database on the network share.
Q: What happens if a technician closes the console or the server hosting the script reboots while it is halfway through scanning 2,000 computers?
The database will not corrupt. The script utilizes an atomic auto-save block. It writes the data in memory to a temporary file (.tmp), validates it, and then instantly swaps it with the live file. The most data we would ever lose from an unexpected termination is the last 49 endpoints scanned. Furthermore, an automatic .bak file is generated at the very start of the execution.
Q: Does this script overwrite existing records? For example, if a user goes on vacation and their PC is offline during the weekly scan, do they disappear from the AD search?
No, the script runs in "Additive Mode." When it loads the existing database at the start, it pulls all historical records into memory. If a PC is offline and fails the ping check, the script simply skips it, but the historical record remains safely in the master array and is saved back to the disk.
Q: Why do we use a composite key (`User-Computer`) instead of just using the Username as the database key?
If we used just the Username, the database would only remember the last computer a user logged into. If a user has a desktop and a laptop, or occasionally logs into a shared kiosk, the composite key allows the database to store and track all of their active devices simultaneously for accurate AD lookups.
