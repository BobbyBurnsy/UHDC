IT Pro Summary

This is a targeted remediation tool designed to resolve stalled Microsoft Endpoint Configuration Manager (MECM/SCCM) deployments. It utilizes WinRM to safely decouple the CcmExec service, enforces a strict thread sleep to ensure local log file locks are released, and spins the service back up to force a fresh machine policy retrieval from the management point.

When to Use This Tool

The Scenario

You just deployed a critical piece of software to a user, but they call the help desk saying, "It's not showing up in my Software Center." Or worse, they see the application, but it is permanently stuck on "Downloading 0%" or "Waiting to install." The standard help desk response is usually, "Please reboot your computer and wait 15 minutes," which disrupts the user's workflow and extends the ticket resolution time.

The Fix

While the user is still on the phone, the technician clicks "Restart SCCM" in the UHDC. The script reaches out and bounces the SMS Agent Host service in the background. The technician tells the user, "Go ahead and click the refresh tab in Software Center." The stuck download immediately resumes, or the missing application instantly populates, allowing the technician to verify the fix and close the ticket on the first call.

[TRAINING] STEP 1: RESTART THE SMS AGENT HOST

WHEN TO USE THIS:
Use this when a user complains that an application in Software Center is stuck on 'Downloading' or 'Waiting to install', or when a newly deployed application is not showing up in their Software Center at all.

WHAT IT DOES:
We are establishing a remote WinRM session to restart the 'CcmExec' (SMS Agent Host) service. We explicitly stop the service, wait 4 seconds to ensure it fully releases its lock on the local SCCM log files (like CAS.log or AppEnforce.log), and then start it again. This forces the agent to wake up and request a fresh machine policy from the management point.

IN-PERSON EQUIVALENT:
If you were physically at the user's desk, you would open Services (services.msc), locate 'SMS Agent Host', right-click it, and select 'Restart'. Alternatively, you would open the Configuration Manager applet in the Control Panel, go to the Actions tab, select 'Machine Policy Retrieval & Evaluation Cycle', and click 'Run Now'.

Invoke-Command -ComputerName $Target -ScriptBlock {
    Stop-Service -Name CcmExec -Force
    Start-Sleep -Seconds 4
    Start-Service -Name CcmExec
}

Technical Execution Pipeline

  1. Pre-Execution Validation: Executes a fast ICMP Ping to verify endpoint connectivity before attempting WinRM connections.
  2. Service Decoupling (WinRM): Establishes a remote PowerShell session over Port 5985 (HTTP) to execute a Stop-Service command against the CcmExec service.
  3. Lock Release Buffer: The script intentionally executes a Start-Sleep -Seconds 4 command. The SCCM agent aggressively writes to dozens of local log files (e.g., C:\Windows\CCM\Logs\). If the service is started too quickly after stopping, it can crash due to lingering file locks. This buffer ensures a clean startup.
  4. Service Restoration (WinRM): Executes a Start-Service command to bring the agent back online, triggering an automatic policy check-in with the Site Server.
  5. Audit Logging: Logs the successful execution to the central UHDC CSV database via SMB (Port 445).

Q&A

Q: Does this reinstall or repair a broken SCCM client?
No. This script simply restarts the existing service. If the SCCM client is fundamentally broken (e.g., WMI repository corruption or missing binaries), the service will fail to start. For deep corruption, you would need to use the UHDC's "Deep Clean" tool to purge the cache, followed by a full client reinstall.
Q: Will the user see anything happen on their screen?
If the user has the Software Center window actively open on their screen, they might see it briefly flash or display a "Loading" spinner for a few seconds while the underlying service reconnects. Otherwise, the process is completely invisible to them.
Q: Why not just use the `Restart-Service` cmdlet instead of Stop, Sleep, and Start?
The native Restart-Service cmdlet executes the stop and start commands almost instantaneously. Because the SCCM agent is a heavy, monolithic service that hooks deeply into WMI and local log files, an instantaneous restart frequently results in the service hanging in a "Starting" state. Manually separating the commands and injecting a 4-second sleep guarantees stability.
