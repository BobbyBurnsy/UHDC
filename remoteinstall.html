IT Pro Summary

This is a dynamic, GUI-driven software deployment engine. It parses a centralized JSON database to build an interactive menu of approved software, and utilizes PsExec to silently push payloads and execute installations on remote endpoints running under the NT AUTHORITY\SYSTEM context. It is designed to bypass the slow sync cycles of traditional deployment tools when immediate remediation is required.

When to Use This Tool

The Scenario

A new hire calls the help desk because they are missing the corporate VPN client or a specialized PDF editor required for their job. You could direct them to the company portal and walk them through the installation, but they don't have local admin rights. Alternatively, you could add them to an SCCM or Intune deployment group, but that might take an hour to sync the policy to their machine while they wait on the phone.

The Fix

The technician clicks "Remote Install." A clean GUI menu pops up populated with standard corporate apps. The technician selects the VPN client and clicks OK. In the background, the script reaches out to the network share, executes the silent installer on the user's machine, and logs the success. The user simply sees the VPN icon appear on their desktop within seconds, allowing the technician to close the ticket and end the call immediately.

Interactive Training Engine

When a junior technician runs this tool with Training Mode enabled, the UHDC pauses execution at each critical phase. It displays a pop-up explaining why the action is necessary, what the in-person equivalent would be, and reveals the exact PowerShell code being executed under the hood. Here is what the trainee sees during a software deployment:

[TRAINING] STEP 1: SILENT REMOTE INSTALLATION

WHEN TO USE THIS:
Use this when a user needs a standard application (like Google Chrome, Adobe Reader, or Zoom) installed, but they do not have local administrator rights, or you want to install it in the background without interrupting their work.

WHAT IT DOES:
We are using PsExec to connect to the target PC as the 'SYSTEM' account. We then execute the installer directly from the network share using 'silent' command-line switches (like /S or /qn). This bypasses UAC prompts and hides the installation wizard from the user.

IN-PERSON EQUIVALENT:
If you were physically at the user's desk, you would open File Explorer, navigate to the network share, double-click the installer, type in your admin credentials when prompted by UAC, and click 'Next' through the installation wizard.

psexec.exe \\$Target -s "$($installer.Path)" $($installer.Args)

Technical Execution Pipeline

  1. Library Parsing (SMB): Bypasses WinRM and connects directly to the central UHDC \Core directory via SMB (Port 445) to read the SoftwareLibrary.json file. It deserializes this data to build the application array.
  2. Interactive UI Generation: Uses a custom WPF XAML window to pause the script and render a searchable, selectable graphical grid for the technician. It leverages the Microsoft.VisualBasic assembly to spawn input boxes if the technician needs to add, delete, or run a custom one-off install.
  3. Payload Execution (RPC/SMB): Dispatches PsExec.exe to the remote endpoint. This establishes a connection over the RPC Endpoint Mapper (Port 135) and executes the remote command. PsExec instructs the remote machine to reach out to the installer's UNC path via SMB (Port 445) and run it using the specified silent arguments (e.g., /S /q).
  4. Audit Logging (SMB): Re-uses the SMB connection (Port 445) to log the specific software deployed and the target endpoint to the central database.

Q&A

Q: Why use PsExec to trigger the installation instead of a standard WinRM Invoke-Command block?
This solves the infamous PowerShell "Double-Hop" problem. If we use WinRM to connect to Machine A, and then tell Machine A to pull an installer from Server B, Windows blocks the secondary network authentication by default. By using PsExec to spawn the process as NT AUTHORITY\SYSTEM, the target machine automatically uses its own Computer Account (DOMAIN\COMPUTER$) to authenticate against the file share, bypassing the double-hop restriction entirely.
Q: Do the network shares hosting our software installers need special permissions for this to work?
Yes. Because PsExec runs the installation as the SYSTEM account, the target endpoint is reaching out to the network share as its machine account, not as the logged-in user or the technician. The UNC share hosting the installers must have "Domain Computers" or "Authenticated Users" added to its Read permissions.
Q: Can the user see the installation happening on their screen?
No. The PsExec command uses the -s switch to run the process under the SYSTEM account, which isolates the execution to Session 0 (the background services session). The user will not see any installation wizards or command prompts pop up, ensuring their workflow is completely uninterrupted.
Q: Where is the SoftwareLibrary.json saved, and what happens if multiple techs try to use it?
It is saved centrally in the $SharedRoot\Core folder. This acts as a unified database. If a Tier 3 technician figures out the perfect silent install arguments for a tricky piece of software, they can add it to the library via the GUI, and every Tier 1 technician will instantly see it in their menu the next time they launch the tool.
