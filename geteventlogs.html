IT Pro Summary

This is a rapid forensic retrieval tool that programmatically extracts event log data from remote endpoints. It features dynamic mode switching: defaulting to a quick-glance critical error check, or scaling up to a deep-scan string match against thousands of logs. It intelligently separates data presentation by keeping the console clean while exporting the full dataset for deep analysis.

When to Use This Tool

The Scenario

A user submits a ticket stating their laptop "randomly blue-screened and restarted 10 minutes ago," or a specific line-of-business application keeps silently crashing without giving the user an error code. Standard protocol means remote-controlling their machine, interrupting their work, and manually digging through the slow Windows Event Viewer GUI.

The Fix

The technician types "bugcheck" or the name of the crashing application into the UHDC keyword box and clicks "Event Logs." The script instantly rips through the last 10,000 System and Application logs, exports all matches to a CSV on the technician's hard drive, and displays the top 15 results directly in the console. The technician identifies the faulting module in seconds, all completely invisibly to the end user.

Interactive Training Engine

When a junior technician runs this tool with Training Mode enabled, the UHDC pauses execution at each critical phase. Because this script has two distinct execution paths (Keyword vs. No Keyword), the training engine dynamically adapts to show the relevant logic. Here is what the trainee sees:

[TRAINING] STEP 1A: QUERY CRITICAL & ERROR LOGS (No Keyword Provided)

WHEN TO USE THIS:
Use this when a user reports unexpected reboots (BSODs), system hangs, or general instability, but doesn't have a specific error code.

WHAT IT DOES:
We are using the 'Get-WinEvent' cmdlet to remotely query the target's System and Application logs. Since no keyword was provided, we are filtering specifically for Level 1 (Critical) and Level 2 (Error) events, pulling the 5 most recent occurrences.

IN-PERSON EQUIVALENT:
Open Event Viewer (eventvwr.msc), expand 'Windows Logs', select 'System', click 'Filter Current Log' on the right pane, and check the boxes for 'Critical' and 'Error'.

Get-WinEvent -ComputerName $Target -FilterHashtable @{LogName=@('System','Application'); Level=@(1,2)} -MaxEvents 5
[TRAINING] STEP 1B: DEEP SCAN FOR KEYWORD (Keyword Provided)

WHEN TO USE THIS:
Use this when troubleshooting a specific failing application (e.g., 'Outlook', 'Teams') or a specific error code provided by the user.

WHAT IT DOES:
We are using 'Get-WinEvent' to pull the last 10,000 events from the System and Application logs, and then piping them into a 'Where-Object' filter to find exact matches for your keyword in either the Message body or the Provider Name.

IN-PERSON EQUIVALENT:
Open Event Viewer (eventvwr.msc), select the 'Application' log, click 'Find...' on the right pane, type your keyword, and click 'Find Next' repeatedly.

Get-WinEvent -ComputerName $Target -LogName 'System','Application' -MaxEvents 10000 | Where-Object { $_.Message -match '$Keyword' -or $_.ProviderName -match '$Keyword' }
[TRAINING] STEP 2: EXPORT AND ANALYZE

WHEN TO USE THIS:
Event logs contain massive blocks of text that are difficult to read in a standard console window. Exporting them to a spreadsheet allows for easy sorting, filtering, and sharing with higher-tier support teams.

WHAT IT DOES:
We are selecting the most relevant properties (Time, ID, Level, Provider, and Message) and exporting them to a local CSV file. We then automatically open File Explorer to highlight the new file for immediate review.

IN-PERSON EQUIVALENT:
In Event Viewer, right-click the filtered log view and select 'Save Filtered Log File As...', save it as a CSV to the Desktop, and open it in Excel.

$logs | Select-Object TimeCreated, Id, LevelDisplayName, ProviderName, LogName, Message | Export-Csv -Path '$ExportPath' -NoTypeInformation
Start-Process explorer.exe -ArgumentList "/select,\`"$ExportPath\`""

Technical Execution Pipeline

  1. Pre-Execution Validation: Executes a fast ICMP Ping to verify endpoint connectivity before attempting RPC connections.
  2. Dynamic Query Execution (RPC): Leverages the native Get-WinEvent cmdlet with the -ComputerName parameter. This executes over Port 135 (RPC) and dynamic high ports to query the remote Windows Event Log service.
  3. Data Serialization & Export: Pipes the raw objects into a Select-Object array to strip out unnecessary XML bloat, then serializes the data to a local, timestamped CSV file (C:\UHDC\Logs\).
  4. Shell Handoff: Executes explorer.exe /select to break out of the PowerShell runspace and visually highlight the newly generated CSV file for the technician.
  5. HUD Formatting: Truncates long error messages to prevent console UI flooding, displaying a neat top-15 preview directly in the dashboard.
  6. Audit Logging: Logs the query action to the central database via SMB on Port 445.

Q&A

Q: What network ports are required for this tool to function?
Unlike our WinRM tools, Get-WinEvent natively relies on Remote Procedure Calls (RPC). This requires Port 135 (RPC Endpoint Mapper) and the standard dynamic RPC port range (49152-65535) to be open between the console and the target. Writing to the audit log requires Port 445 (SMB).
Q: Why does the deep scan limit the search to the last 10,000 events?
Windows Event logs can easily contain millions of entries. Attempting to pull the entire log across the network over RPC can cause severe pipeline lag, memory spikes, and UI lockups. A 10,000-event buffer is the perfect architectural balance—it safely captures recent forensic data while maintaining a sub-10-second response time for the technician.
Q: Why does the console only show 15 logs, but the CSV has all of them?
Console real estate is highly valuable in a dashboard environment. Flooding the UHDC UI with hundreds of multiline error strings would push the command prompt out of view and ruin the user experience. We limit the "Heads Up Display" to the top 15 results and export the full, un-truncated dataset to CSV for deep forensic review.
Q: Does this query the remote 'Security' log too?
No, it specifically targets the 'System' and 'Application' logs. The 'Security' log is heavily restricted, often requiring elevated Domain Admin auditing privileges to read remotely. Furthermore, it is typically flooded with thousands of routine logon/logoff audits, which dramatically slows down query times for standard Tier 1/Tier 2 troubleshooting.
