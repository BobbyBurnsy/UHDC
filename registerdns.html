IT Pro Summary

This is a network routing remediation tool designed to resolve "stale" DNS records. It utilizes PsExec to bypass strict WinRM Kerberos requirements, dispatching a chained sequence of native DOS commands (ipconfig and nbtstat) under the SYSTEM context to force the remote endpoint to update its IP address with the primary Domain Controller.

When to Use This Tool

The Scenario

A user calls the help desk because they cannot access a mapped network drive, or you are trying to remote into their machine using SCCM/MSRA and it keeps failing. The user recently walked from a conference room (where they were on Wi-Fi) back to their desk and plugged into a wired docking station. Their laptop received a new IP address from the DHCP server, but the Domain Controller's DNS cache hasn't updated yet. When you try to connect to their hostname, your computer is trying to route traffic to their old, disconnected Wi-Fi IP address.

The Fix

Instead of asking the user to reboot their computer or walking them through opening a command prompt, the technician clicks "Fix/Reg DNS" in the UHDC. The script pushes through the routing mismatch, forces the user's laptop to flush its old cache, and tells it to actively broadcast its new wired IP address to the Domain Controller. Within a few minutes, network routing is restored and the technician can successfully remote in.

[TRAINING] STEP 1: FLUSH AND REGISTER DNS

WHEN TO USE THIS:
Use this when a computer is turned on and connected to the network, but you cannot connect to it via hostname (e.g., Remote Desktop or File Shares fail). This usually happens when a user switches from Wi-Fi to a wired docking station, giving the laptop a new IP address, but the Domain Controller still has the old IP address cached.

WHAT IT DOES:
We are using PsExec to run a chained command as the SYSTEM account on the remote PC. It flushes the PC's local DNS cache, forces the PC to re-register its current IP address with the Domain Controller's DNS server, and refreshes its NetBIOS names.

IN-PERSON EQUIVALENT:
If you were physically at the user's desk, you would open an elevated Command Prompt and type 'ipconfig /flushdns', press Enter, type 'ipconfig /registerdns', press Enter, and finally type 'nbtstat -RR'.

psexec.exe \\$Target -s cmd /c "ipconfig /flushdns & ipconfig /registerdns & nbtstat -RR"

Technical Execution Pipeline

  1. Intentional Ping Bypass: The script executes a standard ICMP ping. However, unlike other tools, if the ping fails, the script does not abort. Because the tool is designed to fix broken DNS routing, a failed ping is actually the expected behavior (your PC is pinging the wrong IP).
  2. Protocol Selection (RPC/SMB): The script intentionally avoids WinRM (Port 5985). WinRM relies heavily on strict Kerberos authentication, which will instantly fail if the DNS name and IP address do not match. Instead, it deploys PsExec.exe over Port 135 (RPC) and Port 445 (SMB), which are more forgiving and can often resolve the connection via NetBIOS broadcasts.
  3. Command Chaining: To maximize speed and minimize network overhead, the script does not open three separate PsExec sessions. It wraps the three native DOS commands inside a single cmd.exe /c block, separated by the & operator. This executes the entire remediation sequence sequentially in a single payload.
  4. Audit Logging: Logs the successful execution to the central UHDC CSV database via SMB (Port 445).

Q&A

Q: Why does the console say "Target didn't answer ping. Proceeding anyway..."?
If a user's laptop has an IP address of 10.0.0.50, but your Domain Controller thinks it has an IP of 10.0.0.20, your ping command will be sent to .20 and time out. The script recognizes this mismatch is the exact problem you are trying to solve, so it ignores the timeout and attempts to force the connection through anyway.
Q: I clicked the button, but I still can't remote into the PC. Why?
The command executes instantly on the user's laptop, but the laptop then has to send that new IP address to your primary Domain Controller. Depending on your Active Directory site topology, it can take 5 to 10 minutes for that new DNS record to replicate across all the DNS servers in your environment. You may need to run ipconfig /flushdns on your own technician PC to clear your local cache before trying to connect again.
Q: What does `nbtstat -RR` do?
While ipconfig handles standard DNS (Domain Name System) routing, nbtstat -RR drops and refreshes the NetBIOS over TCP/IP name cache. NetBIOS is a legacy broadcast protocol that Windows still heavily relies on for local subnet discovery and file sharing (SMB). Refreshing both ensures all routing avenues are updated.
